# [ADR] 相談一覧の無限ローディング実装方針（MVP）

* **Status**: Accepted
* **Date**: 2026-03-25

---

## 0. このADRで行った意思決定

### 比較した論点

1. スクロール検知を `IntersectionObserver` にするか、`scroll` イベントにするか
2. 取得状態管理を自前（`useReducer`）で持つか、外部ライブラリを導入するか
3. API方式を既存Offsetのまま進めるか、Cursorへ移行するか
4. MVPで仮想スクロールを入れるか
5. 自動連続取得の終端制御を手動ボタンにするか、完全スクロール駆動にするか
6. 重複除去（dedupe）をMVPで必須にするか、後続対応にするか

### 評価観点（Decision Drivers）

- 既存スタックとの親和性・実装速度
- 本番障害リスク（多重リクエスト、競合更新、順序ゆれ）
- バンドルサイズと依存追加の影響
- 保守性と責務分離（Page / Hook / Component / API）
- 将来の拡張容易性（仮想化、Cursor、Queryライブラリ）

### 最終決定

- 検知方式は `IntersectionObserver`（sentinel方式）を採用する
- 状態管理は自前 `useReducer` を採用し、`isFetching` ガードを必須化する
- APIは既存Offset（`page/limit/has_next`）を継続利用する
- MVPは仮想スクロールを導入しない（件数増加時に再評価）
- UXはXライクな完全スクロール駆動（手動「さらに読み込む」は採用しない）
- 401分岐は「API失敗」の子分岐として扱う
- `nextPage = page + 1` を明示的に計算してから取得する
- フロントエンド型を `PaginationMeta` 構造に合わせる
- 重複除去（dedupe）はMVPでは任意（省略する場合は既知リスクとして明記）

### 採用しなかった案

- `scroll` イベント主体の検知
  - 却下理由: 発火頻度と制御負荷が高く、過剰リクエスト事故が起きやすい
- TanStack Query / SWR の即時導入
  - 却下理由: MVP段階では依存追加と学習コストが先行し、変更範囲が広がる
- Cursor Pagination の先行導入
  - 却下理由: API変更コストが高く、現時点ではOffsetで要件を満たせる
- 手動「さらに読み込む」への切替
  - 却下理由: 今回のUX要件（Xライクな連続スクロール）と一致しない

## 1. スクロール検知方式

### 採用: `IntersectionObserver` + sentinel

- `main` のスクロールコンテナ内で sentinel を監視する
- `has_next=true` かつ `isFetching=false` の時のみ次ページ取得を開始する

### 理由

- ブラウザ標準APIで依存追加不要
- `scroll` イベントより無駄な処理が少なく、パフォーマンスと安定性が高い
- 1ページ追加ごとの制御（ガード、失敗分岐、再試行）が明示しやすい

---

## 2. 取得状態管理

### 採用: 自前 `useReducer`（状態機械）

- 代表状態: `initialLoading`, `ready`, `appending`, `initialError`, `appendError`, `end`
- 必須ルール:
  - `isFetching` ガード（多重リクエスト防止）
  - `nextPage = page + 1` を明示
  - 画面離脱時に observer 解放と in-flight request 中断

### 理由

- `useState` の寄せ集めより遷移ミスを防ぎやすい
- ライブラリ導入なしで責務分離を保てる
- MVPに必要な制御を過不足なく実装できる

### 不採用（現時点）: Queryライブラリ導入

- TanStack Query / SWR は有力だが、MVPでの最小変更方針を優先
- 画面横断キャッシュ要件や同期要件が増えた時点で再評価する

---

## 3. API方式（Offset / Cursor）

### 採用: 既存Offsetの継続利用

- 既存契約: `GET /api/consultations?page=...&limit=...`
- 既存レスポンス: `data` + `pagination.has_next`

### 理由

- 既存バックエンドをそのまま活用できる
- 実装範囲をフロント中心に限定できる
- MVPで必要な「連続読み込み」はOffsetで実現可能

### 追加方針

- 並び順の安定化（`createdAt DESC` に tie-breaker を追加）を行う
- フロントの一覧型を `pagination` 契約に合わせる

### 将来再評価

- 投稿頻度増加で重複/欠落が顕在化した場合、Cursor併用を検討する

### セッション切れ時の復元方針（MVP）

- セッション切れ後は一覧位置を復元せず、1ページ目から再開する
- 理由
  - Offset方式では、再ログインまでの間に新規データが挿入されると復元位置の整合性が崩れやすい
  - セッション切れは例外フローであり、MVPでは実装複雑化より安定性を優先する
  - 想定データ量（約500件、20件/ページで約25ページ）では、復元実装を必須化する優先度がまだ高くない
- 再評価条件
  - データ量が増加し、深いページ到達ユーザーの離脱/不満が目立つ場合
  - 「再ログイン後に元の位置へ戻りたい」問い合わせが継続的に発生する場合
  - Cursor方式への移行タイミングで、位置復元要件を正式に再検討する

---

## 4. 描画最適化（仮想スクロール）

### 採用: MVPでは未導入

- まずは通常リストで実装する

### 理由

- 導入コストに対して現時点の効果が限定的
- 先に機能要件（検知・取得・失敗制御）を固める方がリスクが低い

### 再評価条件

- 件数増加でDOM肥大による体感劣化が発生
- 低スペック端末でスクロール性能問題が発生

---

## 5. UX・エラー・A11y方針

### 採用

- Xライクな自動連続読み込み（手動ボタンなし）
- 下部スピナー表示（追加取得中）
- 401はAPI失敗の子分岐で再ログイン導線へ遷移
- 初回エラーと追加エラーを分離表示
- `role="status"` / `aria-live` を利用して状態変化を通知

### 理由

- 期待体験（スクロール継続で読み込み）と一致
- エラー時の行動が明確になり離脱しにくい
- アクセシビリティ要件を最低限満たせる

### 既知の挙動（許容）

- Offline から Online 復帰後、`appendError` 状態で「再試行」を押した際、
  viewport 内に sentinel が残っていると chain-load が続き、最大で1ページ分多く取得する場合がある
  - 例: page1 で失敗復帰時に page2 取得後、そのまま page3 まで連続取得される
- MVPでは以下の理由で許容する
  - 本番想定 `limit=20` のため、追加取得は最大でも概ね +20 件（20 -> 40 件表示）で影響は限定的
  - データ欠落より「通信量増加」が主な影響であり、機能破綻ではない
- 再評価条件
  - APIコスト増、低速回線での体感悪化、過剰取得に関する問い合わせが発生した場合
  - その際は「再試行直後1回のみ chain-load 抑制」または「observer 再開タイミング調整」を導入する

---

## 6. dedupe（MVP）

### 方針

- MVPでは重複除去を必須にしない
- 省略する場合は、以下を必ず明記する
  - 既知リスク: データ変動時に重複表示の可能性
  - 将来対応: API安定ソート + クライアント側dedupe

### 補足

- 安定ソートの追加によりリスクを低減する
- 運用で重複が顕在化した場合は優先度を上げて対応する

## 7. テスト方針

- reducerの状態遷移テストを追加する
- IntersectionObserver のモックを使ったコンポーネントテストを追加する
- 主要E2Eで以下を検証する
  - 下端到達で1ページずつ追加される
  - `isFetching` 中に二重取得されない
  - 401時に失敗分岐から導線が動く

## Summary

| 項目 | 決定内容 | 理由 |
|------|---------|------|
| 検知方式 | IntersectionObserver | 低負荷・依存追加なし・制御しやすい |
| 状態管理 | 自前 `useReducer` | MVPに対して最小コストで堅牢 |
| API方式 | 既存Offset継続 | 既存資産活用・変更範囲最小 |
| 仮想スクロール | MVPでは未導入 | 先に機能安定化を優先 |
| UX | 手動ボタンなしの連続スクロール | 要件（Xライク）に一致 |
| エラー分岐 | 401をAPI失敗の子に配置 | 分岐の責務が明確 |
| 多重取得対策 | `isFetching` ガード必須 | 本番障害予防の要 |
| ページ更新 | `nextPage = page + 1` 明示 | 仕様の明確化・保守性向上 |
| dedupe | MVPでは任意（既知リスク明記） | 速度優先、段階導入 |
