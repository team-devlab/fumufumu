- 議論対象

## 2. 通知フェーズ（共通言語）
※あくまで参考として

1. イベント発生
- 例: 公開アドバイス投稿

2. 通知対象判定
- 誰に通知するか、自己通知除外、対象状態判定

3. 通知レコード保存
- 通知一覧で参照する永続データを保存

4. 配信トリガー
- 同期実行か非同期実行かの分岐

5. 配信実行
- ポーリング / SSE / WebSocket / メール等で通知を届ける

6. クライアント反映
- 通知一覧、未読件数、バッジ更新

7. 既読化
- ユーザー操作で既読更新（冪等）

8. 監視・回復
- 失敗検知、リトライ、重複排除、取りこぼし補完

---

## 3. 比較観点（意思決定の軸）

1. UX速度: どれくらい早く認知させたいか
2. 整合性: 投稿と通知をどこまで一体保証するか
3. 可用性: 通知障害が投稿に波及してよいか
4. 性能: 投稿ピーク時の耐性
5. 実装複雑性: 開発速度とバグリスク
6. 運用複雑性: 監視、リトライ、再実行の難易度
7. 拡張性: 将来の通知チャネル追加余地
8. 調査性: 障害時に原因追跡しやすいか

---

## 4. 配信方式の比較（判断材料）

### A. 同期パターン

- メリット
1. 通知反映が速い
2. 処理経路が単純で追跡しやすい
3. 一体処理の整合性を取りやすい

- デメリット
1. 投稿APIのレイテンシ増加
2. 通知障害の波及リスク
3. 高負荷時のスループット低下リスク

### B. 非同期パターン

- メリット
1. 投稿APIを軽く保ちやすい
2. 障害分離しやすい
3. リトライ/再実行で回復しやすい
4. 将来拡張しやすい

- デメリット
1. 実装と運用が複雑
2. 表示遅延が発生しうる
3. 重複・順序逆転を前提にした設計が必要

---

## 5. 失敗時ポリシー（fail-open / fail-close）

### fail-open

- 定義
- 通知処理が失敗しても、元イベント（例: アドバイス投稿）は成功扱いにする

- 期待効果
1. 投稿の成功率を優先できる
2. 通知障害の影響を限定しやすい

- 懸念
1. 通知遅延/欠落の回復フローが必須
2. 運用監視と再処理の設計が必要

### fail-close

- 定義
- 通知処理が失敗したら、元イベントも失敗扱いにする

- 期待効果
1. 強い整合性を取りやすい
2. データ不一致を減らしやすい

- 懸念
1. 通知障害で投稿失敗が増える
2. 体感品質が下がる可能性

## 9. 用語補足（議論で出た内容）

### レイテンシ増加

- 意味: APIの応答が返るまでの時間が長くなること
- 通知文脈での例: アドバイス投稿APIが通知処理の完了待ちになり、体感が遅くなる

### APIが速い/遅い

- 速い: 混雑時でも応答時間が安定して短い状態
- 遅い: 平均だけでなく、p95/p99（遅い側の分位）で待ち時間が長い状態
- 補足: 体感品質は平均より p95/p99 の影響が大きい

### 可用性の結合

- 意味: 片方の機能障害がもう片方の機能停止につながる状態
- 通知文脈での例: 通知処理障害でアドバイス投稿まで失敗する

### 負荷ピークに弱い

- 意味: 同時アクセス増加時に遅延や失敗率が急増しやすい状態
- 通知文脈での例: 投稿処理内で通知処理も行うと、ピーク時にAPI全体が詰まりやすい

### 再送（リトライ時の重複）

- 意味: クライアント/ネットワーク要因で同一リクエストが再送されること
- リスク: 同じ通知が複数作成される
- 基本対策: 冪等キーやユニーク制約による重複防止

### 非同期の実装・運用複雑性

- 実装複雑性: Outbox/Queue/Worker/Retry等の追加部品が必要
- 運用複雑性: 監視、失敗時の再実行、障害調査の導線が増える

### 重複・順序逆転

- 重複: at-least-once処理で同じイベントが複数回処理される可能性
- 順序逆転: A→Bの順に発生しても処理順がB→Aになる可能性
- 対策: 重複排除、表示時ソート（`created_at + id`）、既読APIの冪等性

### DLQ（Dead Letter Queue）

- 意味: リトライ上限を超えて処理できなかったメッセージの退避先
- 目的: 本線処理を詰まらせず、後で原因調査と再実行を可能にする

### Scheduled Worker（Cloudflare）

- 意味: Cron Triggerで定期起動されるWorker
- 通知文脈での利用: Outboxの未処理イベントを定期的に処理する

## 10. 非同期パターンを選んだ場合の実装パターン（Cloudflare前提）

注意: ここは「実装候補」の整理であり、採用決定ではない。

### パターンA: 最小構成（既存技術中心）

- 想定技術
1. Cloudflare Workers（既存）
2. D1（既存）
3. Cron Trigger / Scheduled Worker（Cloudflare機能追加）

- 構成
1. アドバイス投稿時に `advices` 保存 + `notification_outbox` 保存
2. Scheduled Worker が定期的に outbox を処理
3. `notifications` を作成
4. 失敗は `attempts` / `next_retry_at` / `last_error` を更新
5. 上限超過は `failed` 状態で退避（DLQ相当）

- 特徴
1. 追加サービスが少なく初期コストを抑えやすい
2. 準リアルタイム（Cron間隔依存）

### パターンB: 準リアルタイム構成（Queues追加）

- 想定技術
1. Cloudflare Workers（既存）
2. D1（既存）
3. Cloudflare Queues（追加導入が必要）
4. Queue Consumer Worker（追加）

- 構成
1. アドバイス投稿時にイベントを Outbox か Queue に投入
2. Consumer Worker がイベントを処理
3. `notifications` を作成
4. 失敗時は自動再試行、上限超過はDLQへ

- 特徴
1. Aより低遅延で処理しやすい
2. 運用設計（監視、再実行、障害調査）がより重要

### パターン比較（ざっくり）

1. 開発工数: A < B
2. 運用工数: A < B
3. 低遅延性: A < B
4. 拡張性: A < B
